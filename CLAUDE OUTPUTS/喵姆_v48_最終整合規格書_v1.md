## 0. 文件定位

本文件用於統整三份既有核心文件：

1. `喵姆_v48升級邏輯企劃書_v1.md`
2. `喵姆_v48_Phase0_開發衝刺清單.md`

## 1. 產品定位

喵姆不是傳統看盤網站，不是數據倉庫，也不是即時報價工具。  
喵姆的核心定位是：

> **AI 驅動的台股綜合判斷系統**

它的差異化不是提供更多數字，而是把原始資料消化後，直接輸出：
- BUY / SELL / WAIT
- 信心度
- 理由
- 失效條件
- 後續可擴展的概率輔助層

這個定位在 v48 企劃書中已經定義清楚，也是後續所有架構決策的唯一準繩。

### 1.1 喵姆不做什麼
- 不做即時報價系統
- 不做券商交易端
- 不做社群討論站
- 不做原始基本面資料庫競品

### 1.2 喵姆真正販售的是什麼
- 省下研究時間
- 降低情緒化決策
- 提供可信任的第二意見
- 用 AI 將多源資料轉成一致判斷語言

## 2. 現況診斷：真正的問題不是模型弱，而是資料鏈失真

喵姆現有核心引擎 `miaomu_engine_v2.py` 已具備七維評分能力，並透過裁決引擎輸出 BUY / SELL / WAIT。七維包括：財報真實性、籌碼結構、技術面診斷、市場情緒、惡魔代言人、產業結構、先行指標。

但目前存在三個本質問題：

### 2.1 三個維度長期 fallback
市場情緒、產業結構、先行指標幾乎長期返回 `5/10 neutral`，造成七維中有大量「空氣分數」稀釋真實信號。Cosmos 任務書已明確指出這是核心事實。

### 2.2 資料明明算了，JSON 卻沒正確寫出
v48 與 Phase 0 清單都指出：現況最大的 bug 不是「沒有計算」，而是 `report_to_stock_dict()`、`daily_analysis.json`、前端 key mapping 三者之間對不齊。結果就是：
- 後端有值
- JSON 沒寫對
- 前端讀不到
- 使用者看到的是殘缺或過期狀態

### 2.3 舊值繼承造成假資料可信度
估值欄位如本益比、股價淨值比、殖利率、估值狀態、營收表現，若繼續依賴 `existing.get()` 靜默補值，前端看到的可能不是本次更新結果，而是上一輪殘留。這會直接摧毀產品可信度。這正是 Phase 0 任務 1 要先解的問題。

## 3. 升級總原則

### 3.1 順序不可顛倒
正確順序是：

**Phase 0 → Phase 1 → Phase 2 → Phase 2.5**

不是因為保守，而是因為每一階段都是下一階段的輸入品質前提。v48 企劃書與 Cosmos 任務書都已明確把這個依賴鏈寫出來。

### 3.2 先修資料可信度，再加模型花活
在資料未對齊前：
- 不應先碰 Cosmos 產品上線
- 不應先顯示勝率
- 不應先校準 mu/sigma
- 不應讓概率層反推主裁決

### 3.3 七維主裁決永遠高於輔助層
無論後續加入動態權重或 Cosmos，主裁決仍以七維規則引擎為核心。  
Cosmos 的正確定位是：

> **主裁決之上的概率解釋層，不是主裁決替代品**

## 4. Phase 0：資料可信度修復（立刻開工）

## 4.1 目標
Phase 0 的任務不是讓喵姆變強，而是讓喵姆 **不再說錯話**。  
範圍僅限修復資料流，不新增功能，不碰評分邏輯，不碰 Cosmos，不新增 API。這正是 Phase 0 衝刺清單的明確範圍限制。

## 4.2 任務清單

### Task 1：修 `report_to_stock_dict()` 的估值欄位寫入
修正以下欄位的寫入策略：
- 本益比
- 股價淨值比
- 殖利率
- 估值狀態
- 營收表現

#### 規則
- 有新值：寫新值
- 無新值：寫空字串 / null
- 若真的要沿用舊值：必須明確標記 stale，不得靜默沿用

#### 建議新增欄位
- `valuation_is_stale`
- `data_provenance.generated_at`
- `data_provenance.updater_version`

### Task 2：補齊 `role_outputs[*].raw_data`
將 engine 內部的 `dim.details` 同時映射到：
- `raw_data`
- `details`

目的：
- `raw_data`：供前端統一讀取
- `details`：保留原生語意，方便未來遷移

### Task 3：修前端 key mapping
建立一份唯一 canonical role key map：
- 財報真實性
- 籌碼結構
- 技術面診斷
- 市場情緒
- 惡魔代言人
- 產業結構
- 先行指標

舊名稱只保留一層 fallback，並標示 `legacy` / `TODO remove later`。

### Task 4：加入 `data_provenance`
至少包含：
- `price_date`
- `financial_quarter`
- `chip_date`
- `generated_at`
- `updater_version`

目的不是裝飾，而是建立每支股票輸出的資料可追溯性。這與喵姆「可信第二意見系統」的產品定位直接相關。

## 4.3 驗收標準
至少抽 3 檔股票驗證：

1. `本益比` 不再主要依賴舊值殘留
2. `role_outputs[*].raw_data` 存在
3. `data_provenance` 存在
4. 刪除舊 `daily_analysis.json` 後重建，結果仍正常
5. 前端折疊面板不再因 `raw_data` 缺失出現 `undefined`
6. 雷達圖 7 維名稱完整對齊 canonical keys

## 5. Phase 1：真實資料補強

Phase 1 的目的，是讓目前三個長期 fallback 維度開始擁有真實輸入，並讓估值與營收成為可被後續模型吃下去的第一級資料。這與 v48 的升級邏輯和 Cosmos 任務書的前置假設一致。

### 5.1 必做項目
- 正式接入 `monthly_revenue_growth`
- 正式建立 `pe_percentile_5y`
- 補 `valuation_band` / 估值分位語言
- 讓市場情緒、產業結構、先行指標開始脫離固定 5/10 fallback

### 5.2 原則
- 先求真資料進場，不先追求完美學術建模
- 優先做出可追溯、可驗證、可被主引擎消化的欄位
- 不要先把外部 AI 的軟訊號直接混進核心分數

## 6. Phase 2：動態權重與 regime

v48 企劃書已把 Phase 2 定位為系統靈魂之一：  
不同市場環境下，不應由同一套固定權重解讀所有股票。

### 6.1 實作方向
初版不做五種 regime，先做三種即可：

- `trend`
- `range`
- `high_vol / event_driven`

### 6.2 權重策略
- 先採 rule-based regime 判斷
- 再套 bounded weights
- 權重偏移要有限，避免 regime 誤判時整體翻車

### 6.3 惡魔代言人重構
惡魔代言人不應再被視為與財報、籌碼、技術完全並列的資料維度。  
更合理的定位是：

> **主裁決後的反證層 / consistency checker**

用途：
- 找最弱支柱
- 製造反方論點
- 降低信心度
- 強化失效條件

不建議讓它直接和其他核心資料維度做同權平均。

## 7. Phase 2.5：Cosmos 概率雲引擎

## 7.1 Cosmos 的正確定位
Cosmos 不是預言機，也不是喵姆的新主引擎。  
它的正確角色是：

> **把七維主裁決翻譯成風險語言的概率輔助層**

輸出內容：
- 宇宙勝率
- P10 / P50 / P90 價格區間
- 動態 reason
- 失效條件

Cosmos 任務書已明確要求它必須以真實 `role_outputs`、真實 stock dict、真實 `market_context.regime` 為輸入，且所有係數都必須標註「待校準」。

## 7.2 Cosmos 必須解決的三個問題
Cosmos 任務書已把原始公式的三個致命缺陷寫得很清楚：

### 問題一：fallback 不可被當中性有效訊號
固定 `5/10 neutral` 不應被視為真中性，而應視為資料不足或低資訊狀態。

### 問題二：估值必須進入 `mu`
PE 高低若不影響預期報酬，整個模型就是失真。

### 問題三：成長不應降低波動
高成長與高衰退都應提高 `sigma`，而不是把高成長包裝成更穩定。

## 7.3 Cosmos 設計原則
### fallback 處理
- 從 `plain_text` / `key_finding` 檢測 `資料不足`、`未提供`、`需外部AI補充`
- `score == 5.0` 且 `signal == neutral` 只能作為輔助判斷，不能單獨視為 fallback
- 最終要用平滑品質權重，而不是硬門檻

### mu 原則
`mu` 由有效財報、籌碼、技術、成長動能構成，並受到估值懲罰。  
而且 `mu` 必須允許為負，不能有「再爛都至少正報酬」的地板幻覺。這是任務書特別指出的必要修正。

### sigma 原則
`sigma` 反映不確定性，不反映好壞。  
高波動市場、弱技術、劇烈成長 / 劇烈衰退，都應提高 `sigma`。

## 7.4 Cosmos 的前端表達原則
v48 已提出一個很關鍵的產品風險：  
`勝率 71.3%` 這種數字會給用戶虛假精確度。

### 因此前端規則必須是：
- 不顯示小數點勝率
- 優先顯示 `約 71%`、`65%–75%`、或 `中高機率`
- 必須與失效條件一起出現
- 必須附上「概率模擬輔助層，非投資保證」
- 若與七維主裁決衝突，前端先顯示七維，再顯示 Cosmos 補充說明

## 7.5 Cosmos 何時不該上線
若 Phase 0、1 尚未完成，Cosmos 不應上產品主流程。  
否則它只是把底層資料雜訊包裝成更高級的錯覺。

## 8. 公式校準策略

v48 開放問題已直接問到：  
係數要靠回測、文獻，還是社群反饋？

### 最務實的答案
> **專家初值 + 2018–2025 歷史資料粗校準 + out-of-sample 驗證**

順序如下：

1. 先用專家常識給初值
2. 用 2018–2025 台股資料做粗回測
3. 用樣本外資料驗證穩定性
4. 社群反饋只做產品層微調，不作核心係數主來源

### 不建議
- 一開始就全自動最適化
- 直接照抄學術文獻數值
- 用社群感受決定核心係數

## 9. 產品層級決策

## 9.1 使用者應先看什麼？
v48 已提出：喵姆前端有大量技術圖表，但產品核心其實是判斷。

### 正確資訊層次
1. 主裁決：BUY / SELL / WAIT
2. 信心度與一句話理由
3. 主要矛盾 / 失效條件
4. Cosmos 區間語言（若已啟用）
5. 各維度證據
6. 技術圖表與深度資料

也就是說，**先看判斷，再看證據，再看圖表**。

## 9.2 七維與 Cosmos 衝突時怎麼辦？
答案很明確：

> **七維優先，Cosmos 輔助**

原因：
- 七維是主裁決引擎
- Cosmos 是翻譯層
- 翻譯層不能推翻主引擎

這不只是產品美學，而是系統一致性問題。v48 已把這題列為開放問題，但本整合文件正式定案採此立場。

## 10. 交付給其他 AI / 工程代理的硬性規則

以下規則不可違反：

### 10.1 Phase 0 規則
- 不改 `miaomu_engine_v2` 評分邏輯
- 不新增分析維度
- 不新增 API 呼叫
- 不先碰 Cosmos / Monte Carlo
- 只修資料流與前端對齊

### 10.2 Cosmos 規則
- 所有係數都標註 `待校準`
- 不得假裝已完成歷史回測
- `mu` 必須可為負
- `sigma` 必須讓高成長與高衰退都提高波動
- 不得把 fallback 直接當中性有效訊號
- 不得讓 Cosmos 取代主裁決

### 10.3 輸出規則
其他 AI 若接這份文件，產出必須至少包含：
1. Phase 0 patch 任務實作
2. 驗收結果
3. 若做 Cosmos，必須給完整 Python 程式碼，不可只給偽代碼
4. 若寫 PRD，必須清楚分開：
   - 主裁決層
   - 概率輔助層
   - 視覺呈現層
   - 風險揭露層

## 11. 最終決策

### 11.1 現在立刻要做什麼
**立刻執行 Phase 0。**

### 11.2 現在絕對不要做什麼
- 不要先上 Cosmos
- 不要先顯示勝率
- 不要先校準 mu/sigma
- 不要先讓外部 AI 反推主裁決

### 11.3 為什麼
因為目前最值錢的升級不是更高級的模型，而是：

> **把喵姆從「可能算對但顯示失真」修成「真的可信、真的對齊、真的可追溯」**

只有這一步完成後，後面的 Phase 1、Phase 2、Phase 2.5 才有商業意義。
