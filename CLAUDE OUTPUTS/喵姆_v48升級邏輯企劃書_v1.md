## 第一章：喵姆是什麼？產品定位

### 1.1 一句話定位

喵姆是一個 **AI 驅動的台股綜合判斷系統**，而非傳統的行情看板或數據展示工具。

核心差異化：市面上的看盤軟體（TradingView、Yahoo Finance、Win 投資）是「給你數據，你自己判斷」；喵姆是「AI 把數據消化完，直接告訴你 BUY / SELL / WAIT，並且解釋為什麼」。

### 1.2 喵姆不做什麼

- 不做即時報價（那是券商 App 的工作）
- 不做回測平台（雖然有回測功能，但這不是核心賣點）
- 不做社群討論（不是 PTT 股版或投資社群）
- 不做基本面資料庫（不跟 Goodinfo 或公開資訊觀測站競爭數據完整性）

### 1.3 喵姆的目標用戶

對股市有興趣但不是全職交易員的人。他們：
- 沒時間每天看盤三小時
- 看得懂「本益比」但不會自己算合理價區間
- 想要一個「可信任的第二意見」，而不是一個數據堆砌的儀表板
- 願意為「省下研究時間」和「減少情緒化決策」付出代價

### 2.2 現有七維度評分機制詳解

| 維度 | 輸入資料 | 評分邏輯 | 資料狀態 |
|------|---------|---------|---------|
| 財報真實性 | 2季損益表+資產負債表+現金流量表 | 現金流品質比、AR膨脹率、存貨異常、業外佔比 | ✅ 有資料，正常運作 |
| 籌碼結構 | 外資/融資/股價5日變動 | 12組共振矩陣 + 密度 + 大戶集中度 | ✅ 有資料，正常運作 |
| 技術面診斷 | MA/RSI/MACD/布林/支撐壓力 | 指標方向加減分 | ✅ 有資料，正常運作 |
| 市場情緒 | sentiment dict | 恐懼貪婪、新聞情緒、散戶信心 | ❌ 永遠 fallback 5/10 |
| 惡魔代言人 | 前四維結果 | 反向質疑最弱環節 | ✅ 正常運作（依賴前四維品質） |
| 產業結構 | industry dict | 護城河、定價權、敘事崩塌偵測 | ❌ 永遠 fallback 5/10 |
| 先行指標 | leading dict | 供應鏈、專利、共識偏差 | ❌ 永遠 fallback 5/10 |

**關鍵發現：七維中有三維（情緒、產業、先行）永遠返回 5/10 neutral。**

這代表裁決引擎實際上只靠「財報 + 籌碼 + 技術 + 惡魔代言人」四個維度在運作。當四維中有三維全是 5/10 neutral 時，裁決幾乎必然是 WAIT + LOW confidence——這解釋了為什麼台積電目前顯示「觀望（訊號混亂）」。

### 2.3 裁決引擎邏輯

```
一票否決（最高優先級）：
  IF 財報真實性 < 3 → SELL 或 WAIT
  IF 籌碼 danger + anomaly ≥ 8 → SELL
  IF 產業敘事崩塌 → SELL

正常裁決（bullish/bearish 計數）：
  4 個基礎維度中：
    ≥4 bullish → BUY HIGH
    ≥3 bullish → BUY MEDIUM
    ≥3 bearish → SELL MEDIUM
    ≥4 bearish → SELL HIGH
    其他 → WAIT LOW

深度調整：
  惡魔代言人 < 4 且 BUY → 降級為 WAIT
  產業結構 ≥ 8 且 BUY → 信心升 HIGH
  先行指標 bearish 且 BUY → 信心降 LOW
```

**問題**：因為三個維度永遠 5/10 neutral，它們不會被計為 bullish 也不會被計為 bearish。這使得 bullish 計數最多只有 3（財報+籌碼+技術全部偏多），很難達到 HIGH confidence。裁決引擎的設計沒問題，但它被餵了 43% 的空氣數據。

## 第三章：已發現的系統性問題

### 問題一：資料斷層——engine 算了但 JSON 沒寫

`report_to_stock_dict()` 中，以下欄位的寫入方式是 `existing.get(...)`：

```python
"本益比": existing.get("本益比", ""),      # ← 從舊資料繼承
"股價淨值比": existing.get("股價淨值比", ""),
"殖利率": existing.get("殖利率", ""),
"估值狀態": existing.get("估值狀態", ""),
"營收表現": existing.get("營收表現", ""),
```

engine 的 `FinancialForensicsV2` 內部有計算 net_income、revenue 等數字，但這些算完的結果只參與了評分，沒有回寫到最終 JSON。前端想顯示本益比時，拿到的是空字串或上次殘留的舊值。

**影響**：前端基本面面板裡的數字要麼是空的，要麼是過期的。用戶看到一個「觀望」判斷，但點進去看基本面，什麼數字都沒有——無法追溯判斷依據。

### 問題二：JSON 結構與前端期待不匹配

engine v2 輸出的 `role_outputs` 結構：
```json
{"role_name": "財報真實性", "score": 5, "signal": "neutral", "plain_text": "...", "key_finding": "...", "mekak": "..."}
```

前端讀取時有兩個問題：

**(a) 名稱不一致**：前端同時搜尋舊版名稱（`技術分析官`、`籌碼分析官`）和新版名稱（`技術面診斷`、`籌碼結構`），用 `||` fallback。但其他維度（如 `財報真實性`）在前端可能沒有對應的讀取邏輯。

**(b) 缺少 `raw_data` / `role_data` 子結構**：前端多處期待 `ro[i].raw_data`（技術面的 MA60、籌碼面的詳細數據），但 engine v2 只輸出 `details` dict，且 `stock_updater` 在轉換時沒有把 `details` 映射為 `raw_data`。

結果：前端能讀到分數，但嘗試從 `raw_data` 取詳細數據時拿到 undefined，導致面板部分區塊顯示空白或 fallback。

### 問題三：已建好的能力沒接進管線

`data_fetcher.py` 有以下方法但未被 `stock_updater.py` 的主流程呼叫：
- `fetch_monthly_revenue()`——月營收
- 可能還有其他 fetch 方法（需進一步確認）

這代表系統有能力取得月營收資料，但沒有人把開關打開。

### 問題四：三個維度永遠是空氣

情緒、產業結構、先行指標三個維度需要的 `sentiment`、`industry`、`leading` 資料，在 `build_engine_input()` 中永遠傳入空 dict：

```python
return {
    "financial": financial_data if financial_data else {},
    "chip": chip_data,
    "technical": technical,
    "sentiment": {},      # ← 永遠空
    "industry": {},       # ← 永遠空
    "leading": {},        # ← 永遠空
}
```

這不是 bug，是設計上的「先佔位，後填充」策略。但實際效果是：七維分析裡有三維永遠返回中性 5 分，拉低了整體判斷品質，也使得「惡魔代言人」維度缺乏足夠的對比資料。

## 第四章：v48 升級提案——不是加模組，是修管線

### 4.0 設計哲學：從「展示數據」到「交付判斷」

> **原則一**：喵姆的每一個數字都必須服務於最終的 BUY/SELL/WAIT 判斷。如果一個數字只是「有了比較好看」但不參與判斷邏輯，就不應該進入主界面。
>
> **原則二**：AI 的判斷必須可追溯。用戶看到「WAIT」，應該能一路點回去看到「因為估值偏高（本益比 30 > 歷史中位數 22）+ 籌碼偏空（外資連賣 5 日）+ 成長強（營收年增 25%）→ 三者衝突所以觀望」。
>
> **原則三**：權重必須是動態的，不是固定公式。戰爭爆發時，「地緣政治風險」的權重應該從 0 飆升到主導地位。央行意外升息時，「利率敏感度」應該壓過技術面。這不是人為調參數，而是讓 AI 根據「市場狀態」自動調整。

### 4.1 Phase 0：修復現有管線（不加新東西，先把已有的接好）

**目標**：讓 engine 算出的數字正確寫入 JSON，前端正確讀取。

**具體修改**：

1. **`report_to_stock_dict()` 重寫估值欄位**：從 `financial_data` 和 `price_history` 計算本益比、殖利率、股價淨值比，而非從 existing 繼承。

2. **`role_outputs` key 對齊**：統一使用 engine 輸出的結構，前端側修改讀取邏輯來配合。

3. **加入 `data_provenance`**：每個寫入 JSON 的數字旁邊標註資料來源和時間，例如：
```json
{
  "本益比": 30.1,
  "本益比_source": "FinMind Q3 2025",
  "本益比_date": "2025-11-14"
}
```

**預期效果**：修完後，現有的七維分析面板、基本面面板、籌碼面板會從「半空」變成「有完整數據」，而且數字和判斷可以互相印證。

### 4.2 Phase 1：填充空氣維度——月營收 + 估值區間

**目標**：讓七維中的「空氣維度」開始有真實資料。

**1A. 月營收趨勢進入 engine**

- 在 `stock_updater.py` 的 `update_single()` 中呼叫 `fetch_monthly_revenue()`
- 計算衍生指標：月增率、年增率、連續成長月數、累計年增率
- 這些指標有兩個去處：
  - (a) 送進 `financial` dict，強化「財報真實性」維度的評分（目前只看季報，加入月營收能提高時效性）
  - (b) 新增到 `build_engine_input()` 的 `sentiment` 或新增一個 `growth` 子項，讓「市場情緒」維度有部分真實數據

**1B. 本益比河流圖（估值區間）**

- 用近 5 年 EPS + 當前股價，計算合理價區間
- 區間定義：
  - 便宜：EPS × 歷史最低 PE
  - 合理：EPS × 歷史中位數 PE
  - 偏高：EPS × 歷史 75 百分位 PE
  - 昂貴：EPS × 歷史最高 PE
- 這個計算結果進入 JSON 的新欄位 `valuation_band`
- 同時寫入 `估值狀態` 欄位（取代目前的空字串或人工標注）
- 參與「財報真實性」或新增「估值面」維度的評分

### 4.3 Phase 2：動態權重系統——這是喵姆真正的靈魂

> Kirin 的原話：「更厲害的軟體應該是動態的考慮，而不是靜態的。根據發生什麼事情，某些東西的比重就會隨之變多。」

**這個想法有充分的學術和工程依據。**

在量化金融中，這叫做 **Regime-Dependent Weighting**（狀態依賴權重），或者更通俗地說，**Market Regime Switching**。核心概念是：

1. 市場不是一種狀態，至少有 3-5 種：正常上漲、正常下跌、高波動恐慌、低波動盤整、事件驅動（戰爭/政策/黑天鵝）
2. 不同狀態下，同一個指標的預測力完全不同。例如：在低波動盤整期，技術面的均線交叉非常有效；但在戰爭爆發時，均線完全沒用，因為股價的驅動力來自地緣政治而非技術形態
3. 聰明的系統會先判斷「現在是什麼市場狀態」，再根據狀態調整各維度的權重

**喵姆已經有 `regime_classifier.py` 這個檔案**（在 scripts/ 目錄下），代表之前就有人想做這件事。

**具體設計**：

```
[市場狀態偵測器] RegimeDetector
  輸入：
    - 大盤指數近 20 日波動率（VIX 替代：ATR/price）
    - 大盤漲跌幅（近 5 日、20 日）
    - 市場寬度（上漲家數/下跌家數）
    - 外部事件標記（從新聞情緒或人工標記）

  輸出：5 種狀態之一
    BULL_TREND    → 穩定上漲
    BEAR_TREND    → 穩定下跌
    HIGH_VOL      → 高波動（恐慌或狂熱）
    RANGE_BOUND   → 盤整
    EVENT_DRIVEN  → 事件驅動（戰爭/政策/黑天鵝）
```

```
[動態權重矩陣]

                    BULL   BEAR   HIGH_VOL  RANGE  EVENT
  財報真實性         20%    25%     10%      25%     5%
  籌碼結構          25%    20%     15%      20%    10%
  技術面診斷         25%    20%     10%      30%     5%
  市場情緒          10%    15%     30%      10%    20%
  估值面(新)        15%    15%     10%      10%     5%
  事件影響(新)       5%     5%     25%       5%    55%
```

注意看 EVENT_DRIVEN 那一欄：「事件影響」權重從平時的 5% 飆升到 55%，而技術面從 25% 降到 5%。這就是你說的「發生戰爭的時候，它就換另外一組判斷基準」。

**這不是拍腦袋的數字**，而是基於一個原理：當某個因素成為股價的主要驅動力時，其他因素的邊際解釋力就會下降。技術分析在正常市況下有效，是因為市場參與者按照技術面在交易；但當戰爭爆發，大家不看技術面，所以技術面的預測力趨近於零。

**實作方式**：

1. `VerdictEngineV2.judge()` 目前的邏輯是「數 bullish/bearish 個數」——這是等權重。改為「加權分數」：

```python
# 現在（等權重）：
base_bullish = sum(1 for d in base_dims if d.signal == "bullish")

# 改為（動態權重）：
regime = self.detect_regime(market_context)
weights = self.WEIGHT_MATRIX[regime]
weighted_score = sum(d.score * weights[d.name] for d in all_dims)
```

2. 市場狀態從 `daily_analysis.json` 的 `market_context.regime` 讀取（這個欄位已經存在）。

### 4.3.5 Phase 2.5：Cosmos 概率雲引擎——把分數轉成人話

#### 概念

七維分析輸出的是「1-10 分 + bullish/bearish」，專業但不直觀。Cosmos 引擎把這些分數轉化成：

- **宇宙勝率**：一年後上漲機率（例如 71.3%）
- **價格區間**：10%-90% 分位數（例如 1620-2740 元）
- **動態失效條件**：什麼情況下這個預測作廢

方法：用七維分數動態計算 GBM（Geometric Brownian Motion）的 mu（漂移率）和 sigma（波動率），然後跑 2000 次 Monte Carlo 模擬。

**這不是取代七維判斷，而是把判斷翻譯成普通人的語言。**

#### 公式設計（含外部 AI 提案與本方驗證）

外部 AI 提出的 mu/sigma 映射公式：

```python
growth_momentum = (fin_score/10 * 0.45) + (monthly_growth/100 * 0.35)
chip_confirmation = (chip_score/10) * 0.40
tech_momentum = (tech_score/10) * 0.25
mu = (growth_momentum + chip_confirmation + tech_momentum) * 0.22
```

**本方用實際數字驗證後發現三個問題（已用 Python 跑過數據）**：

**問題 A：fallback 排除是斷崖式而非平滑**

修正版用 `if fin_score == 5.0: fin_weight = 0` 做排除。但 5.0 和 5.1 之間存在斷崖：

| fin_score | mu |
|-----------|------|
| 5.0 | 0.1061（被排除）|
| 5.1 | 0.1566（+0.0505 跳躍）|
| 8.0 | 0.1854（只比 5.1 多 0.0287）|

5.0→5.1 的 mu 變化（0.0505）比 5.1→8.0 的變化（0.0287）還大。這代表 engine 的 fallback 判斷如果有一點點誤差（例如 4.9 vs 5.0），整個 Monte Carlo 的結果會劇烈翻轉。**建議改用平滑衰減函數取代硬門檻。**

**問題 B：估值因子方向正確但幅度需校準**

修正版加入 `valuation_penalty = (pe_percentile - 50) / 100 * 0.25`，效果：

| PE 百分位 | mu | 勝率 |
|----------|------|------|
| 10th（很便宜）| 0.2853 | 77.3% |
| 50th（中位數）| 0.1854 | 67.4% |
| 90th（很貴）| 0.0854 | 54.6% |

方向正確（越貴 → 勝率越低）。但「PE 90th 的股票仍有 54.6% 勝率」是否合理？歷史數據顯示，PE 在 90th 百分位以上的股票，後一年的平均報酬率接近市場平均甚至略低。54.6% 可能偏樂觀。**需要用台股歷史數據校準 valuation_penalty 的係數。**

**問題 C：mu 地板 0.02 導致垃圾股也有 ~46% 勝率**

修正版 `mu = max(0.02, ...)` 意味著公式認為任何股票至少有 2% 年化報酬。驗證結果：

```
全部最差（fin=1, chip=1, tech=1, growth=-30%, PE=99th）:
  mu=0.0200 (地板), sigma=0.3370
  勝率=45.5%
```

一支各方面最差的股票仍然有 45.5% 勝率，這不合理。**建議 mu 的地板改為 -0.10（允許負報酬預期），或者在 mu < 0 時直接觸發 SELL 判斷而不進入 Monte Carlo。**

#### 修正後的建議公式（待校準）

```python
def calculate_mu_sigma(fin, chip, tech, monthly_growth, pe_pct,
                       data_quality_flags: dict):
    """
    data_quality_flags: {"financial": True/False, "chip": True/False, ...}
    True = 有真實資料, False = fallback
    """
    # 平滑衰減取代硬門檻（問題 A 修正）
    def quality_weight(score, has_data):
        if not has_data:
            return 0.0  # 無資料 = 不參與
        # 分數越接近 5（中性），可信度越低
        confidence = abs(score/10 - 0.5) * 2  # 0~1，5分=0，1或10分=1
        return (score/10) * confidence

    fin_w = quality_weight(fin, data_quality_flags.get("financial", False))
    chip_w = quality_weight(chip, data_quality_flags.get("chip", True))
    tech_w = quality_weight(tech, data_quality_flags.get("tech", True))

    growth_momentum = (fin_w * 0.45) + (monthly_growth / 100 * 0.35)
    chip_signal = chip_w * 0.40
    tech_signal = tech_w * 0.25

    raw_mu = (growth_momentum + chip_signal + tech_signal) * 0.22

    # 估值打折（問題 B）
    valuation_penalty = (pe_pct - 50) / 100 * 0.25

    # 允許負 mu（問題 C 修正）
    mu = raw_mu - valuation_penalty
    mu = max(-0.10, min(mu, 0.35))  # 允許到 -10% 年化

    # sigma
    base_vol = 0.22
    tech_penalty = (1 - tech/10) * 0.18
    growth_vol = abs(monthly_growth) / 100 * 0.12  # 用絕對值：衰退也增加波動
    sigma = base_vol + tech_penalty + growth_vol
    sigma = max(0.15, min(sigma, 0.50))

    return mu, sigma
```

## 第五章：評分機制重設計——牽一髮而動全身

### 5.1 現有問題

目前的最終評分是所有維度的簡單平均：

```python
scores = [d.score for d in all_dims if d.score is not None]
result["評分"] = round(sum(scores) / len(scores), 1)
```

7 個維度，3 個永遠是 5 分，另外 4 個就算全部 8 分，平均也只有 (8×4 + 5×3) / 7 ≈ 6.7 分。用戶看到 6.7 分，感覺「還好」，但實際上如果只看有效維度，8 分是很強的多頭信號。

**簡單平均 + 空氣數據 = 永遠溫吞的結論。**

### 5.2 新評分邏輯

```
最終評分 = Σ (維度分數 × 動態權重) / Σ 動態權重
            ↑                      ↑
       只計算有真實資料的維度    權重由市場狀態決定
```

如果某個維度是 fallback（資料不足），它不參與加權計算，權重直接為 0。這避免了「空氣維度稀釋真實信號」的問題。

### 5.3 裁決引擎重設計

不再用 bullish/bearish 計數，改為加權綜合分：

```
加權總分 = Σ (維度分數 × 動態權重)

裁決規則：
  加權總分 ≥ 7.5  → BUY
  加權總分 ≤ 3.5  → SELL
  其他            → WAIT

信心度規則：
  所有有效維度方向一致（全 bullish 或全 bearish）→ HIGH
  有效維度中 ≥70% 方向一致                        → MEDIUM
  方向不一致（衝突）                               → LOW

一票否決仍然保留（財報嚴重異常、籌碼出貨陷阱）。
```

### 5.4 矛盾偵測與表達

當有效維度之間出現嚴重矛盾（例如「成長面 9/10 但估值面 2/10」），AI 不應該輸出一個「平均」的結論，而應該輸出：

> **WAIT — 信號衝突**
> 成長面強勁（營收年增 45%），但估值已在歷史極端值（PE 35 > 95th percentile）。
> 這代表：市場已經提前反映成長預期。
> 觀察點：下一季營收年增率是否能維持 40% 以上，若能則估值有支撐。

這比「6.5 分，觀望」有用一百倍。

## 第六章：動態權重的科學依據

### 6.1 學術基礎

- **Markov Regime Switching Models** (Hamilton, 1989)：市場在不同狀態之間切換，每個狀態有不同的統計特性
- **Conditional Factor Models**：因子的解釋力在不同市場環境下不同（Fama-French 因子在熊市的表現和牛市不同）
- **Adaptive Markets Hypothesis** (Andrew Lo, 2004)：市場效率不是固定的，而是隨環境演化的

### 6.2 實務案例

- **橋水基金 (Bridgewater) 的 All Weather Strategy**：根據經濟環境（成長/通膨的升/降組合）動態調整資產配置權重
- **Man Group 的 AHL**：根據波動率狀態切換趨勢跟蹤策略的參數
- **台灣的元大投信**：其 AI 選股 ETF 會根據市場狀態切換選股因子的權重

### 6.3 喵姆的簡化版實作

喵姆不需要做到橋水的複雜度。最小可行版本：

1. 用 `market_context.regime` 判斷當前狀態（這個功能已經存在，`regime_classifier.py` 已經在計算）
2. 硬編碼 5×7 的權重矩陣（5 種狀態 × 7 個維度）
3. `VerdictEngineV2.judge()` 在裁決前先查詢當前狀態，套用對應權重
4. 未來可以用歷史回測來自動優化權重矩陣

## 第八章：風險與限制——誠實面對

### 8.1 系統性風險

- **AI 判斷的準確率問題**：喵姆不是黑盒子預測系統，它的判斷是基於規則的（加權評分），不是基於機器學習預測的。好處是可解釋，壞處是不會「學習」新模式。
- **資料延遲**：月營收每月 10 號公布，季報更慢。喵姆的基本面判斷永遠是「事後」而非「預測」。技術面和籌碼面是當日資料，兩者時間頻率不同。
- **FinMind API 依賴**：單一資料源。如果 FinMind 掛了或改 API，整個系統停擺。

### 8.2 動態權重的風險

- **過度擬合**：如果用歷史回測來優化權重，很容易過度擬合過去的市場狀態。建議先用「專家直覺 + 常識」設定初始權重，再用回測驗證，而非直接用回測結果。
- **狀態誤判**：如果 RegimeDetector 判斷錯誤（例如把高波動恐慌判斷成正常下跌），權重分配會錯誤。需要有 fallback：如果狀態判斷信心不足，回退到等權重。
- **黑天鵝**：真正的黑天鵝事件（COVID、311）是無法被歷史模式捕捉的。喵姆應該在這種情況下誠實輸出「WAIT — 市場狀態超出模型範圍，建議人工判斷」。

### 8.3 產品定位風險

- **法律風險**：台灣《證券投資信託及顧問法》規定，提供個股投資建議需要執照。喵姆如果面向公眾，需要加上免責聲明：「本系統為資訊工具，非投資建議」。
- **用戶過度依賴**：如果用戶把 BUY/SELL 當作絕對指令，虧錢了怎麼辦？喵姆的判斷必須永遠附帶「失效條件」和「信心度」，提醒用戶這不是確定的事。
